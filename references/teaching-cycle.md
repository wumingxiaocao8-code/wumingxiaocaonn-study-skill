# 教学循环详细步骤

## 完整教学循环（学习模式 / 精读模式）

每个模块必须经过以下 7 个阶段。未通过测试的模块回到 TEACH 阶段（只补缺口，不全讲）。

### 1. DIAGNOSE（诊断，预期时长：3-5 分钟）

**目的**：探测用户对当前模块内容的已有理解，避免浪费时间在已掌握内容上。

**操作**：
- 出 3-5 个快速问题，覆盖模块的核心概念
- 问题从易到难：先问"X 是什么"，再问"X 和 Y 有什么区别"
- 每题给用户 1-2 分钟回答
- 记录正确/错误/模糊的分布

**根据诊断结果调整教学**：
- 全部正确 + 有深度：建议跳过此模块或加速测试
- 部分正确：标记错题为教学重点
- 全部错误：从最基础的概念开始，不跳过任何部分

### 2. TEACH（教学，预期时长：5-10 分钟）

**原则**：
- 只讲诊断中暴露的缺口 + 核心难点
- 每次教学不超过 3 个关键点（防止认知过载）
- 每个关键点配：解释 → 类比 → 例子 → 反例

**教学结构**：
1. 指出目标："今天我们要理解的是 X，它在实际中的作用是 Y"
2. 核心讲解：定义 → 机制 → 应用
3. 类比/例子：用一个生活化的类比
4. 对比：X 和容易混淆的 Y 有什么区别
5. 反例：什么情况下 X 不成立？为什么？

**避免**：
- 信息倾倒（一次塞太多概念）
- 照本宣科（不要逐字复述书本）
- 跳过类比（没有类比的教学很难形成深层记忆）

### 3. QUESTION（苏格拉底提问，预期时长：3-5 分钟）

**目的**：检验理解深度，引导用户自己发现深层逻辑。

**最少 2 层追问**：

第 1 层（理解验证）：
- "你能用自己的话说一遍吗？"
- "给我一个书上没有的例子。"

第 2 层（深度推理）：
- "如果 X 的前提条件变了，结论还成立吗？"
- "持相反观点的人会怎么反驳？"
- "这和[之前学过的概念]有什么关联？"

第 3 层（可选，用于概念类高目标等级）：
- "这个理论的局限是什么？"
- "你能设计一个反例来测试这个结论吗？"

**提问技巧**：
- 问题要具体，不要"你怎么看"
- 给用户思考时间
- 追问不等于为难——是帮助用户把模糊理解变清晰

### 4. TEST（综合测试，预期时长：5-15 分钟）

**出题规则**（按内容类型）：

每道题出题时即标注题型标记：
- **定义**：问"是什么"、"有哪些"。结论对即满分，不触发过程审计。
- **机制**：问"为什么"、"如何传导"、"因果链"。必须执行过程审计。
- **对比**：问"区别"、"优劣"、"适用条件"。结论对 + 关键维度对即满分。

**概念类**（经济学、哲学、理论）：
- 至少 1 道解释题："请解释 X，给出你自己的例子"
- 至少 1 道对比题："X 和 Y 有什么区别？各举一个场景"
- 至少 1 道机制/传导题（因果链或传递路径）
- 可选：场景分析题："在情境 Z 中，X 如何应用？"

**实操类**（Python、量化交易）：
- 至少 1 道代码题："请写一个函数实现 X"
- 至少 1 道解释题："你写的代码中，为什么第 Y 行要那样写？"
- 可选：边界条件题："如果输入是空列表，你的代码会怎样？"

**评分**：
- 每题 1-5 分（对应 0-5 掌握度）
- 总分取平均
- **定义/列举题**：结论对即满分（5 分），不触发过程审计
- **机制/传导题**：按过程审计规则（结论 + 推理链二维评分），答对后追加 1 个机制追问（不计分但记入推理备注）
- **对比/分析题**：结论对 + 关键维度覆盖完整 → 满分；遗漏重要维度 → 扣 1 分
- 解释题同时侧重"是否用自己的话 + 是否有自己的例子"
- 代码题侧重"功能正确 + 能解释原理"

**评分记录格式**：

```
| 题号 | 题型标记 | 类型 | 结论 | 推理质量 | 扣分点 | 需复测 |
|------|----------|------|------|----------|--------|--------|
| 1 | 定义 | 解释题 | ✅ | — | — | 否 |
| 2 | 机制 | 传导题 | ✅ | 完整 | — | 否 |
| 3 | 机制 | 因果题 | ✅ | 跳步 | 遗漏"信心危机→抛售"环节 | 是 |
```

### 5. CORRECT（纠错，预期时长：3-5 分钟）

**对每个错误执行**：

1. **定位错误**：具体错在哪一步/哪个概念
2. **分类错误类型**：
   - 记忆错误（记错了术语或事实）
   - 概念混淆（把 X 当成 Y）
   - 逻辑跳跃（跳过了必要的推理步骤）
   - 前提错误（基于错误的前提推导）
3. **找到根因**：为什么会错？是概念没理解还是推理方式有问题？
4. **微型教学**：针对这个具体错误，换一个角度重新解释
5. **提供锚点**：给出一个容易记住的类比/口诀/反例
6. **变体验证**：出一道变体题，确认已纠正

### 6. REVIEW（模块总结，预期时长：2-3 分钟）

**用户任务**：
- "用你自己的话，总结这个模块你学到了什么"
- "这个模块和你之前学过的什么内容有关联？"
- "你觉得最难理解的部分是什么？"

**AI 任务**：
- 补充用户没有提到的关键点
- 指出潜在的概念关联（包括跨书关联）
- **快速检查机制覆盖**：回顾本模块的核心机制，确认教学和测试已覆盖全部。如有遗漏，补充 1 道机制题或指出缺口
- 在 session 记录末尾写入一行 "本模块覆盖机制：A / B / C"
- 预告下一模块的大致内容

### 7. RECORD（记录）

写入学习档案：
1. 新建或更新 `sessions/YYYY-MM-DD.md`
2. 更新 `module-plan.md`（状态、掌握度、复习日期）
3. 如有错题 → 追加到 `mistakes.md`
4. 如有新概念关联 → 更新 `concept-map.md`
5. 更新 `review-queue.md`（根据复习间隔：1 天/3 天/7 天/30 天）
6. 更新 `book-index.md`
7. 更新 `global-index.md`
8. 更新 `current-state.md`

## 概念微循环（快速答疑模式）

```
判断已有理解 → 解释 → 追问 1-2 层 → 必要时记录
```

**判断已有理解**（1 个问题）：
- "关于 X，你目前了解到什么程度？"

**解释**：
- 控制在 1-2 个关键点，配 1 个类比
- 不展开，不旁征博引

**追问 1-2 层**（不是必须 2 层）：
- "这样解释清楚吗？你再用自己的话说一遍？"
- 如果用户能清晰复述 → 结束
- 如果模糊 → 再补 1 个解释

**"必要时记录"的判断标准**：
- 该概念属于用户当前学习书籍的关键概念 → 追加到 concept-map.md
- 用户在追问中表现出深度兴趣 → 追加到 current-state.md 的未解决问题
- 否则不记录

## 复习模式循环

```
读取待复习项 → 重测 → 判定 → 更新掌握度 + 复习队列
```

1. 读取 review-queue.md 中的到期项
2. 对每个待复习模块出 2-3 道测试题
3. 全部正确 → 掌握度不变或提升，下次复习间隔延长一档
4. 有错误 → 掌握度下调，进入纠错流程，下次复习间隔缩短
5. 更新 review-queue.md

复习间隔：
- 1 天后 → 3 天后 → 7 天后 → 30 天后 → 归档（不再主动提醒）
- 如果复习时出错，降一档重新开始
